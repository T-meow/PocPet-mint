# PocPet 1.5.0 全量打包与发布计划

## 目标

- 将应用版本统一提升到 `1.5.0`。
- 本机生成并验证 Windows x64/x86、Android arm64/ARMv7 和 Web 包。
- 推送 `main` 与 `v1.5.0` 标签，由 GitHub Actions 生成 macOS/Linux 等全量产物并发布 GitHub Release。
- 通过现有 GitHub Pages workflow 部署 `dist/`。

## 范围与非目标

- 影响版本元数据、构建产物、GitHub Actions 和 GitHub Pages。
- 本地 `release/` 最终只保留 `1.5.0` 的 APK、EXE 和 Web ZIP；macOS/Linux 产物仅由 CI 构建。
- Android 包沿用 debug keystore，仅用于测试分发；不配置商店签名。
- 不修改应用功能、存档格式或用户数据结构。

## 实施阶段

1. 同步 `package.json`、lockfile、Tauri 与 Cargo 的版本号。
2. 运行前端构建以及项目现有的专项检查。
3. 构建 Windows x64/x86、Android arm64/ARMv7 和 Web 产物，并校验文件名、大小和签名。
4. 清理 `release/` 中非本次 APK、EXE、Web ZIP 的内容。
5. 提交并推送 `main`，创建并推送 `v1.5.0` 标签。
6. 等待 GitHub Pages 与全量 Release workflow 完成，核对部署地址和 Release 产物。

## 影响文件与系统

- `package.json`、`package-lock.json`
- `src-tauri/Cargo.toml`、`src-tauri/Cargo.lock`
- `src-tauri/tauri.conf.json`
- 本地 `release/` 交付目录
- GitHub Actions、GitHub Pages、GitHub Release

## 兼容性与风险

- `1.5.0` 只改变应用版本元数据，不改变存档 schema，现有存档保持兼容。
- Android `versionCode` 由版本号生成，`1.5.0` 对应 `10500`。
- macOS/Linux 只能在对应 GitHub runner 上验证；CI 失败时不发布不完整 Release。
- Pages 部署使用现有 workflow，部署内容与本机 `npm run build` 结果一致。

## 验证计划

- 运行 `npm run build` 和 `npm run check:*` 专项检查。
- 核对五类本地产物：x64/x86 EXE、arm64/ARMv7 APK、Web ZIP。
- 使用 `apksigner verify` 校验两个 APK，检查 Web ZIP 内容与 EXE 内嵌资源。
- 核对版本字段、Git 状态、远端提交与 `v1.5.0` 标签。
- 等待并确认 GitHub Pages 和 PocPet CI and Release workflows 成功。

## 假设

- `origin` 指向 `T-meow/PocPet`，默认分支为 `main`。
- 当前工作树内已有的未推送提交属于本次远端发布范围。
- GitHub 仓库已配置 Pages 与 Actions 所需权限。
