# Pocpet-Mint

[Chinese](README.md)

Pocpet-Mint is a desktop and mobile virtual pet app built around companionship, Pomodoro sessions, items, pet status, and replaceable pet mods. The project is built with Tauri, React, TypeScript, and Rust, and aims to provide a lightweight, customizable, cross-platform personal desktop pet experience.

May virtual companionship soothe lonely souls.

The project is still in an early stage. APIs, save formats, and mod specifications will continue to evolve. Issues, suggestions, and ideas are welcome.

## Open Source And AI Participation

Pocpet-Mint is a personal experimental project. Most of the code is generated by AI programming assistants or created with AI assistance, then manually selected, integrated, debugged, and released by the maintainer. The code structure, gameplay values, and interface details will continue to change according to personal needs. There is no promise of a stable roadmap or long-term compatibility with every downstream development direction.

The collaboration workflow of this repository is also AI-first. If you submit a PR, code review and merge decisions will most likely also be assisted by AI, while the maintainer only provides limited direction confirmation and release control. Because of that, it is recommended to use issues for bug reports, reproduction steps, feature requests, or design discussion. If you want to maintain your own long-term changes, gameplay branch, commercial packaging, or release cadence, it is better to fork this repository and replace assets, adjust rules, and organize your own release workflow in your own branch. Changes that differ significantly from Pocpet-Mint's current design direction are also better developed independently in a fork.

## Feature Overview

- Virtual pet status: hunger, cleanliness, mood, energy, health, and other basic states.
- Items and shop: stable core item IDs, with support for food, gifts, care items, and utility items.
- Pomodoro timer: work and rest flows affect pet activity feedback.
- Mods: import zip packages to replace pet images, item images, default name, default birthday, display text, and favorite food.
- Save import and export: back up and restore current data with a versioned text format.
- Basic multilingual support: currently includes Chinese and English text.

## Local Development

Requirements:

- Node.js 18 or newer
- npm
- Rust stable
- System dependencies required by Tauri 2
- Android builds require Android SDK, NDK, and a Java/Gradle environment.

Install dependencies:

```bash
npm install
```

Start the frontend dev server:

```bash
npm run dev
```

Start Tauri development mode:

```bash
npm run tauri:dev
```

Frontend build check:

```bash
npm run build
```

## Packaging

Windows portable exe:

```powershell
npm.cmd run package:win:portable
```

Web deployment package:

```powershell
npm.cmd run package:web
```

Android arm64 APK:

```powershell
npm.cmd run package:android:arm64
```

By default, the Android packaging script rebuilds the Tauri Android arm64 native library and embedded frontend resources, then generates, aligns, debug-signs, and verifies the APK. The web deployment package compresses the current `dist/` directory into `release/pp-Mint<version>-web.zip`.

macOS and Linux must be built on their corresponding systems:

```bash
npm run package:desktop
```

macOS artifacts are mainly `.dmg` / `.app`. Linux artifacts are mainly `.AppImage` / `.deb`, targeting Ubuntu-compatible environments first. Domestic Linux distributions can usually test the AppImage first, while Debian/Ubuntu-based distributions can test the deb package.

## GitHub Actions Auto Build And Release

The repository includes `.github/workflows/release.yml`. Default behavior:

- Pull requests and normal branch pushes: only run `npm run build` for frontend and TypeScript build checks.
- Pushing a `v*` tag or manually running the workflow: build daily test packages.
- If the current version is a semver `x.y.0`, or if `full_build` is selected during manual workflow dispatch: additionally build Web, macOS, and Linux artifacts.
- Pushing a `v*` tag: automatically create a GitHub Release and upload artifacts after the build finishes.

Daily test packages:

- Windows x64: `pp-Mint<version>.exe`
- Android arm64 debug-signed APK: `pp-Mint<version>.apk`

Full builds additionally produce:

- Web deployment package: `pp-Mint<version>-web.zip`
- macOS: `pp-Mint<version>-mac.dmg`
- Ubuntu/Linux: `pp-Mint<version>-ubuntu.AppImage`; if the runner also generates a deb package, `pp-Mint<version>-ubuntu.deb` is kept as well.

Android packages use a temporary debug keystore generated on the CI runner by default and are only suitable for test distribution. Official store signing requires separate signing keys and a release process.

## Gitee Auto Build Template

The repository also keeps `.gitee/workflows/release.yml` as a Gitee auto build template. It is designed to build the following artifacts on push or tag events:

- Windows: exe
- Android: APK
- macOS: dmg/app
- Linux: AppImage/deb, preferably Ubuntu-compatible and also convenient for testing on domestic Linux distributions.

To enable automatic publishing to Gitee Release, configure the following secrets in the Gitee repository's pipeline/Actions settings:

- `GITEE_TOKEN`: a personal access token with permission to create releases and upload assets.
- `GITEE_OWNER`: the repository owner user name or organization name, for example `ferrisM`.
- `GITEE_REPO`: the repository name, for example `poc-pet`.

macOS builds require a macOS runner. If Gitee does not provide a matching hosted runner, use a self-hosted runner. Linux builds are recommended on Ubuntu 22.04 or newer. Android builds require the runner to have Android SDK/NDK installed, or the pipeline needs to install them.

Gitee pipeline capabilities, runner labels, and Release APIs may vary by account or enterprise configuration. If the platform syntax or runner labels differ from this template, adjust `.gitee/workflows/release.yml` according to the YAML generated by the current Gitee pipeline interface.

## Mod Creation

Guides:

- `docs/mod制作指南.md`
- `docs/mod-guide.md`

New mods are recommended to use `schemaVersion: 2`. They can replace pet images, item images, default name, default birthday, display text, favorite food, good-ending CG, and can add namespaced safe custom items. Mods do not expose holiday reward pools, core gameplay values, save rules, or Pomodoro rules.

## Save Compatibility

The project keeps compatibility with the internal legacy save format `pocpet-mint.pet.v1`. External import and export use a versioned text format. During import, the time baseline is reset to avoid immediate offline decay or Pomodoro auto-settlement after restoring an old backup.
